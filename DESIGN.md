## 1. 제품 정의

### 1.1 한 줄 정의
끓는 물을 부은 뒤 **수십 초의 짧은 시간** 동안, 화면을 보든 안 보든 "지금 따라낼 때"임을 자연스럽게 깨닫게 해 주는 모바일 차 우림 타이머.

### 1.2 핵심 사용자 흐름
1. 앱을 연다. 홈 화면 아이콘 또는 브라우저에서 세로 화면으로 실행한다.
2. 차를 고른다. 차 캐러셀의 좌우 버튼으로 넘기면 권장 시간과 권장 온도 토스트가 반영된다.
3. 우림 도구를 고른다. 히어로 양쪽 화살표 또는 스와이프로 전환한다.
4. 큰 시간 조정 영역에서 슬라이더와 `-`/`+` 버튼으로 시간을 조정한다.
5. `우리기 시작`을 누르면 물 붓는 연출 후 우림이 시작된다.
6. 우림 중 찻물 색, 찻잎, 김, 글로우, 링, 숫자, 예고음이 진행감을 만든다.
7. 완료 시 종소리, 플래시, 도구별 완료 메시지가 나온다.
8. 다음 우림이면 `다음 泡 우리기`를 누른다. 표일배는 먼저 `버튼 눌러 내리기`로 아래 잔 배수를 진행한다.

### 1.3 설계 원칙
- **도구 앱처럼 정돈되되, 차 도구처럼 따뜻해야 한다.** UI는 절제하고, 감각적 분위기는 중앙 도구·탕색·김·소리에서 만든다.
- **숫자는 보조, 감각 신호가 주역이다.** 탕색·링·글로우·증기·예고음·플래시를 겹쳐 어느 하나를 놓쳐도 전달되게 한다.
- **차 우리기의 종료는 0초가 아니라 분리/따르기다.** 모든 도구에는 실제 완료 행동이 있다.
- **고정 도구는 이미지 자산, 변하는 현상은 코드다.** 기물 형태를 코드 path로 억지로 그리지 않고, PNG 자산 위에 액체·잎·김·링만 SVG로 합성한다.
- **수십 초 스케일에 맞춘 긴장 곡선**을 쓴다. 마지막 3-5초에 임박 신호를 집중시킨다.
- **엄지 영역 우선**이다. 주요 조작은 하단에 두고 모든 터치 타깃은 44px 이상이어야 한다.
- **오프라인·즉시 실행**을 목표로 한다. 현재 PWA 앱 셸은 캐시되지만 폰트는 아직 CDN 의존이다.

- 통합 전 `CLAUDE.md`에는 개완 뚜껑 열림이 적용된 것으로 적혀 있었으나, 현재 `index.html` 기준으로는 미작동이라 정정했다.

- 통합 전 README의 "가로 스와이프로 차 전환" 표현은 현재 구현과 달랐다. 스와이프는 도구 히어로에만 적용되어 있어 README를 정정했다.

### 4.3 도구별 신호 적응
- 유리 도구인 개완과 표일배는 찻물 색, 수면, 찻잎을 직접 보여준다.
- 티팟과 다호는 내부 탕색이 보이지 않으므로 앰버 글로우, 김, 링, 소리로 진행감을 만든다.
- 머그는 윗면 수면색만 보여주고 링은 쓰지 않는다.
- 표일배는 윗 챔버에서 우린 뒤 완료 상태에서 메인 버튼으로 아래 잔 배수를 수행한다.
- 현재 링은 `round:true`인 티팟·다호에 표시된다. 개완은 `ringGeom()` 기본값이 남아 있지만 `round:false`라 실제 표시는 꺼져 있다.

## 5. 우림 연출

### 5.1 긴장 곡선

| 구간 | t 범위 | 표현 | 소리 |
|---|---|---|---|
| A. 우러나기 | 0-0.5 | 색이 초반 빠르게 오르고 잎이 펴지기 시작 | 무음 |
| B. 무르익음 | 0.5-0.85 | 색은 완만히 포화, 김과 글로우 안정 | 무음 |
| C. 임박 | 마지막 3-5초 | 숫자·메시지·링·글로우가 강조 | 예고음 + 매초 tick |
| 완료 | t >= 1 | 완료 메시지, 플래시, 완료 상태 고정 | 종소리 |

### 5.2 구현된 연출
- 색 곡선은 `easeInOut` 대신 추출 포화형 `sat(t)`를 쓴다.
- 개완 액체는 path 기반 형태, 세로 그라데이션, meniscus, 벽면 wash를 쓴다.
- 표일배 아래 잔은 `rect + clip`에서 파라메트릭 `path`로 바뀌었다.
- 찻잎은 차종별 프로필, 잎별 delay, sway, flutter, pour 난류를 가진다.
- 수면은 `shimmerSurf()`로 미세하게 일렁이며 임박 구간에서 진폭이 커진다.
- 불투명 도구는 `ambientGlow`와 `steamPorts()`로 진행감을 만든다.
- pour 구간은 0.9초이며 물줄기와 착지 잔물결이 표시된다.

### 5.3 아직 구현되지 않은 연출
- 개완 완료 시 뚜껑 림라이트 이동과 틈 하이라이트. 현재 `lidTarget`과 `lidLift`는 계산되지만 `#g-lid`에 쓰이지 않는다.
- 회차별 탕색 점감. 현재 5泡도 1泡와 같은 `peak` 농도를 사용한다.
- 도구 전환 전이. 현재는 `display:none`과 표시 전환만 사용한다.

## 7. 기술 아키텍처

### 7.1 파일 구성

```text
/index.html              # 앱 전체: HTML + CSS + JS
/manifest.webmanifest    # PWA 설치 메타
/sw.js                   # 서비스워커, 앱 셸 cache-first
/icons/*.png             # PWA 아이콘
/assets/vessels/*.png    # 도구 이미지
/assets/vessels/source/  # 원본/중간 자산
```

### 7.2 PWA와 캐시
- manifest는 `display: standalone`, `orientation: portrait`, `theme_color: #120d09`를 사용한다.
- `sw.js`는 cache-first 앱 셸 전략이다.
- 코드 수정 후 브라우저에서 옛 코드가 보이면 서비스워커 등록 해제와 캐시 삭제가 필요하다.
- 앱 아이콘 PNG는 이미 존재하므로 "아이콘 추가"는 더 이상 TODO가 아니다.
- 폰트는 아직 Google Fonts CDN에 의존하므로 완전 오프라인 기준에서는 미완료다.

### 7.3 타이머 정확도
- 우림 종료는 `performance.now()` 기반 절대 종료 시각으로 계산한다.
- `visibilitychange`로 포그라운드 복귀 시 Wake Lock을 재획득한다.
- 사운드는 현재 rAF 흐름에서 호출된다. 백그라운드 정확도를 높이려면 WebAudio clock 예약 또는 Worker tick이 다음 단계다.

### 7.4 상태 머신

```text
idle
  └─ startBrew()
       └─ pour (0.9s)
            └─ steep
                 └─ finishSteep()
                      ├─ done: 일반 도구 → 다음 泡 우리기
                      └─ done: 표일배 → 버튼 눌러 내리기 → drainPiao() → 다음 泡 우리기
```

## 8. 접근성·모션·터치

### 8.1 반영된 기준
- `#timeLabel`은 평소 `aria-live="polite"`, 완료 시 `assertive`로 전환한다.
- 헤더의 선택 차·회차 정보는 시각적으로 숨긴 live region에서 보조기술에만 전달한다.
- 주요 버튼과 아이콘 버튼은 44px 이상 터치 타깃을 가진다.
- `prefers-reduced-motion`이면 urgent animation과 플래시 animation을 끈다.
- vessel/tea 셰브런 대비는 `--ink-dim`으로 보강되었다.
- 색만으로 완료를 알리지 않고 숫자, 메시지, 소리, 플래시를 같이 쓴다.

### 8.2 남은 접근성 점검
- 차 캐러셀 스와이프가 구현되면 버튼 접근성과 충돌하지 않아야 한다.
- 완료 플래시 대비는 reduced-motion 외에도 완료 메시지 가독성을 실제 화면으로 재검증해야 한다.
- 폰트 셀프호스팅 후 fallback과 글자폭 변화가 모바일 레이아웃을 흔들지 않는지 확인한다.

## 12. 검증 절차

### 12.1 서비스워커 캐시 초기화

```js
(async()=>{
  for(const r of await navigator.serviceWorker.getRegistrations()) await r.unregister();
  for(const k of await caches.keys()) await caches.delete(k);
  location.reload();
  return "reloaded";
})()
```

### 12.2 rAF 기반 시간 진행 검증

헤드리스 미리보기는 비활성 상태에서 `requestAnimationFrame`이 멈출 수 있다. 시간 진행은 eval 내부에서 rAF 루프를 돌려 확인한다.

```js
(async()=>{
  const slider = document.getElementById("slider");
  const brew = document.getElementById("brew");
  const time = document.getElementById("time");
  slider.value = 8;
  slider.dispatchEvent(new Event("input"));
  brew.click();
  const samples = [];
  for(let i = 0; i < 600; i++){
    await new Promise(r => requestAnimationFrame(r));
    samples.push(time.textContent);
    if(brew.textContent.includes("다음") || brew.textContent.includes("버튼")) break;
  }
  return { frames: samples.length, last: samples.at(-1) };
})()
```

### 12.3 시각 작업 판정
- 좌표 오버레이는 clip과 요소 위치를 확인하는 데만 쓴다.
- 좌표가 맞아도 물성은 틀릴 수 있다. 렌더 이미지를 처음 보는 사용자 시선으로 다시 평가한다.
- 같은 결함을 좌표만 바꿔 반복 수정하면 primitive가 틀렸다는 신호다. 액체는 영역 clip이 아니라 형태 path가 책임져야 한다.
